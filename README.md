# foundry-and-litellm

Use **Microsoft Foundry – Bring Your Own Model (BYOM)** with **[LiteLLM](https://github.com/BerriAI/litellm)** as the AI gateway, deployed on **Azure Container Apps** with Terraform.

Foundry's [bring your own model](https://learn.microsoft.com/en-us/azure/foundry/agents/how-to/ai-gateway) capability lets Foundry Agent Service call models hosted behind your own gateway. LiteLLM is a great fit: it exposes an OpenAI-compatible API in front of virtually any provider (Azure OpenAI, OpenAI, Anthropic, Bedrock, self-hosted, …). Foundry connects to it via a **ModelGateway** connection.

## Architecture

Two Foundry instances demonstrate true *bring your own model*: the **agents**
Foundry runs agents but owns no model; the **models** Foundry hosts the model.
LiteLLM brokers between them and authenticates to the models Foundry with a
managed identity (no keys).

```
  +----------------------+   ModelGateway connection (ApiKey)
  | Foundry "agents"     |   POST /chat/completions          +------------------------+
  | account + project    | --------------------------------> | LiteLLM gateway        |
  | (runs agents, BYOM)  |      Bearer <master key>          | (Azure Container Apps) |
  +----------------------+                                   +-----------+------------+
                                                                         | Entra ID token
                                                                         | (user-assigned MI,
                                                                         |  Cognitive Services
                                                                         |  OpenAI User)
                                                                         v
                                                             +------------------------+
                                                             | Foundry "models"       |
                                                             | account                |
                                                             | gpt-5.4-mini deployment |
                                                             +------------------------+
```

What Terraform provisions (`terraform/`):

| Resource | Purpose |
| --- | --- |
| Foundry "models" account (`AIServices`, local auth disabled) + `gpt-5.4-mini` deployment | Hosts the actual model |
| Foundry "agents" account + project (`AIServices`) | Runs agents; owns no model |
| User-assigned managed identity + `Cognitive Services OpenAI User` on the models account | Lets LiteLLM call the model with Entra ID, keyless |
| Container Apps environment + LiteLLM container app | The OpenAI-compatible AI gateway, publicly reachable over HTTPS |
| Log Analytics workspace | Container Apps logs |
| ModelGateway connection (on the agents project) | Wires the agents project to LiteLLM with the master key |

The LiteLLM master key is generated by Terraform and used both as the gateway
auth key (clients -> LiteLLM) and as the credential stored on the agents-project
connection. The LiteLLM -> models-Foundry hop is keyless and uses managed identity.

## Prerequisites

- Azure subscription with quota for `gpt-5.4-mini` (Global Standard) and access to Microsoft Foundry.
- [Terraform](https://developer.hashicorp.com/terraform/install) >= 1.6 and [Azure CLI](https://learn.microsoft.com/cli/azure/install-azure-cli).
- `az login` to the target subscription.

## Deploy

```bash
cd terraform
cp env.sample .env
# edit .env: set TF_VAR_subscription_id and TF_VAR_gh_repo (owner/repo)
source .env

terraform init
terraform apply
```

Key outputs:

| Output | Description |
| --- | --- |
| `litellm_endpoint` | Public base URL of the LiteLLM gateway |
| `litellm_master_key` | Gateway auth key (sensitive) |
| `foundry_project_endpoint` | Foundry project endpoint for the agents SDK |
| `foundry_model_deployment_name` | `litellm-gateway/gpt-5.4-mini` — use this as the agent model |

## Test the gateway directly

```bash
cd samples
# Create samples/.env from Terraform outputs
{
  echo "LITELLM_ENDPOINT=$(terraform -chdir=../terraform output -raw litellm_endpoint)"
  echo "LITELLM_MASTER_KEY=$(terraform -chdir=../terraform output -raw litellm_master_key)"
} > .env

./test_gateway.sh
```

`test_gateway.sh` loads `samples/.env` by default (see `samples/.env.sample`).
Pass a different path as the first argument, or set `LITELLM_ENDPOINT` /
`LITELLM_MASTER_KEY` in the environment to skip the file.

> **Note:** This deployment runs LiteLLM standalone (no database), so the admin
> **UI is not available** — logging in returns `Not connected to DB!`. Use the
> proxy API with the master key as shown above. Enabling the UI requires
> attaching a Postgres database and setting `DATABASE_URL`.

## Run a Foundry agent through the gateway

The only thing that makes the agent "BYOM" is the model name: `<connection-name>/<model-name>`.

```bash
cd samples
export FOUNDRY_PROJECT_ENDPOINT="$(terraform -chdir=../terraform output -raw foundry_project_endpoint)"
export FOUNDRY_MODEL_DEPLOYMENT_NAME="$(terraform -chdir=../terraform output -raw foundry_model_deployment_name)"

pip install -r requirements.txt
python run_agent.py
```

## Adding more models

1. Add the upstream model to `terraform/config/litellm-config.yaml.tftpl` under `model_list` (and any credentials it needs).
2. Add a matching entry to `gateway_models` in `terraform/locals.tf` so Foundry advertises it.
3. `terraform apply`.

The connection uses a **static model list** with `deploymentInPath = "false"`, so Foundry calls `{endpoint}/chat/completions` and selects the model via the request body — exactly LiteLLM's contract. To use dynamic discovery instead (LiteLLM exposes `GET /v1/models`), replace `models` in the connection metadata with a `modelDiscovery` block. See the [ModelGateway connection reference](https://github.com/microsoft-foundry/foundry-samples/blob/main/infrastructure/infrastructure-setup-bicep/01-connections/model-gateway/ModelGateway-Connection-Objects.md).

## Logging

By default the proxy logs **only the request and response payload** for each
call, as a single structured JSON line. This is driven by a small custom
callback (`terraform/config/litellm_request_logger.py`) wired in via
`litellm_settings.callbacks`, which keeps the noisy framework loggers quiet.

```bash
# Tail just the request/response entries from the running container app
az containerapp logs show -g <rg> -n aca-litellm-<suffix> --follow \
  | grep request_response
```

In Log Analytics:

```kusto
ContainerAppConsoleLogs_CL
| where Log_s has "litellm.request_response"
| extend p = parse_json(tostring(parse_json(Log_s).message))
| project TimeGenerated, model=p.request.model, messages=p.request.messages, response=p.response
```

Set `TF_VAR_litellm_log_level=DEBUG` (variable `litellm_log_level`) and re-apply
for full framework tracing when you need to debug the proxy itself.
